# 《项目开发执行手册：GitHub 仓库知识理解与问答助手》

**版本**：V1.0  
**用途**：本手册不是给老师看的项目介绍，而是给 AI 大模型 / 开发执行者直接施工用的实现说明。  
**目标**：让执行者按本文档逐步完成一个**面向公开 GitHub 仓库的知识理解与问答助手**，并以**轻量桌面应用 + 本地后端服务**作为交付形态。  

---

# 0. 强制前提（开工前必须接受）

以下前提是本项目的硬约束，执行时不得擅自扩大范围：

1. **只做公开 GitHub 仓库**  
   MVP 不处理私有仓库权限接入，不做组织权限系统。

2. **只做单机单用户本地优先架构**  
   桌面端和后端默认运行在用户本机，不依赖云数据库、云向量库、远程任务队列。

3. **只做单仓库问答上下文**  
   用户可以管理多个仓库，但一次问答只能绑定一个仓库，禁止跨仓库混合上下文。

4. **只做只读分析**  
   系统不能执行仓库脚本、不能自动安装依赖、不能运行未知代码、不能自动改代码或提交 PR。

5. **不能把整个仓库一次性塞给模型**  
   必须采用“先扫描 → 建 Repo Map → 建索引 → 按问题检索 → 局部回答”的路线。

6. **必须返回证据**  
   最终答案不能只有自然语言，至少要返回文件路径、片段位置、来源类型。

7. **前后端保持轻量**  
   禁止微服务拆分、禁止 K8s、禁止 Redis/Celery、禁止 PostgreSQL/Elasticsearch/Milvus 这类重型基础设施。

8. **MVP 先支持主流仓库结构**  
   优先覆盖 Python、JavaScript、TypeScript 项目；其他语言先按通用文本模式降级处理。

9. **多 Agent 只保留弱编排思想**  
   不做重型多 Agent 自治系统。可以内部保留几个角色模块，但运行时由单主流程统一控制。

---

# 1. 项目目标（实现导向表述）

本项目最终要实现一个本地桌面应用，用户可以：

1. 选择或登记一个公开 GitHub 仓库（以本地克隆目录为主）
2. 对该仓库执行首次扫描与索引
3. 查看仓库概览、关键模块、技术栈、推荐阅读路径
4. 就该仓库提出自然语言问题
5. 获得带证据的回答
6. 获取有限度的改进建议
7. 在大仓库场景下仍然通过分层扫描与检索控制 token 消耗

这个系统的产品定位是：

> **把公开 GitHub 仓库接入为临时代码知识库，帮助用户快速认识项目、围绕项目提问，并获得结构化理解结果。**

# 2. 严格范围定义

## 2.1 首版 MVP 必做范围

以下能力必须优先进入首个可运行版本。首版目标不是一次性做全，而是先打通“仓库接入 → 索引 → 检索 → 带证据问答”的最小闭环：

1. 添加本地仓库（本地 Git 克隆目录）
2. 校验仓库目录并扫描文件清单
3. 解析 README、Markdown、Python、JavaScript、TypeScript 和常见配置文件
4. 生成可检索 chunk，并保存来源文件、行号、类型等证据信息
5. 建立 SQLite 元数据表和 SQLite FTS 关键词索引
6. 接入 Chroma 本地向量索引，形成关键词 + 向量的混合检索
7. 生成基础 Repo Map 与仓库概览
8. 提供问答接口，并返回答案、证据、置信度和下一步建议
9. 提供三栏桌面界面：仓库列表 / 问答区 / 证据区
10. 所有关键失败返回中文错误提示

## 2.2 首版后增强范围

以下能力有实用价值，但不阻塞首版闭环，建议在 MVP 跑通后再做：

1. 公开 GitHub URL 自动 clone 到本地 `data/repos`
2. 可选配置 GitHub Token，用于补充仓库元数据与 GitHub 链接
3. 增量同步流程
4. 文件上下文查看
5. 问题模板
6. 重建索引 / 清空索引
7. 日志与诊断页
8. 设置页
9. 会话隔离与历史裁剪
10. 建议生成与规则检测

## 2.3 首版禁止范围

以下内容首版禁止实现：

1. 登录 / 注册 / 权限中心 / 团队共享
2. 云端部署 / 微服务 / K8s / 消息队列集群
3. 浏览器插件 / 移动端 App / SaaS 控制台
4. 自动改代码 / 自动提 PR / 自动发布
5. 企业级统一知识库 / 跨仓库全局知识图谱
6. 语音输入 / 图片理解 / OCR / 多模态
7. GitHub Issue / PR 双向写回
8. 整仓长上下文直接问答
9. 复杂自治多 Agent
10. 企业级安全审计承诺

# 3. 技术选型（固定建议）

如果没有充分理由，不要修改以下技术选型：

## 3.1 桌面端
- **Electron + React + TypeScript + Vite**
- UI 组件库：**Mantine** 或 **Ant Design** 二选一，禁止混用
- 状态管理：**Zustand**
- 前端与后端通信：**localhost HTTP(JSON)**
- 如需进度推送：**SSE**，首版不需要 WebSocket

## 3.2 本地后端
- **Python 3.11**
- **FastAPI + Uvicorn**
- **Pydantic v2** 用于请求、响应、配置校验

## 3.3 存储层
- **SQLite**：存仓库记录、任务记录、会话记录、索引元数据、日志索引
- **FAISS CPU** 或 **Chroma 本地持久化**：做向量检索，二选一  
  默认建议：**Chroma**，实现简单

## 3.4 检索层
- **关键词检索 + 向量检索混合**
- 关键词部分优先使用轻量本地方案（如 Whoosh 或 SQLite FTS）
- 不能只依赖向量召回

## 3.5 模型层
- 本地生成模型接入：**Ollama HTTP API**
- 本地嵌入模型接入：**Ollama embeddings** 或 **sentence-transformers**
- 模型名称必须配置化，不能写死在业务逻辑中

## 3.6 代码与文档解析
- Markdown：保留标题层级、列表、代码块边界
- Python：优先用 `ast`
- JS/TS：优先用 tree-sitter 或轻量结构解析
- JSON/YAML/TOML：按配置对象处理

# 6. 代码规范要求（硬性要求，必须执行）

## 6.1 注释规范（关键逻辑优先，避免注释拖慢开发）

### 文件级注释
每个核心 `.py` 文件顶部必须先写一段中文注释，至少说明两件事：

1. 这个文件是干什么的
2. 这个文件在整个项目框架里扮演什么角色

示例要求：

```python
"""
这个文件负责仓库扫描。
它在整个框架里扮演“知识接入入口”的角色，负责把本地仓库变成后续可索引的数据来源。
"""
```

### 函数 / 类定义前的注释块
核心函数、核心类、复杂算法和容易被误用的接口前，必须写中文注释块，且建议包含以下 4 项：

1. **作用**：这段代码是干什么的
2. **原理**：它为什么这么实现，不是只说“做了什么”，而是说“为什么这样做”
3. **参数与返回值**：每个参数是什么意思，返回什么
4. **如何调用**：给一个最简单的调用示例

示例要求：

```python
# 作用：扫描仓库目录，收集后续索引需要的文件信息。
# 原理：先递归遍历目录，再结合忽略规则过滤掉无关文件，这样可以减少后续解析和向量化开销。
# 参数：
# - repo_path: 仓库根目录路径
# 返回值：
# - 返回一个文件记录列表，每条记录包含路径、类型、大小等基础信息
# 如何调用：
# files = scan_repository_files("D:/projects/example-repo")
def scan_repository_files(repo_path: str) -> list[dict]:
    ...
```

不强制给所有简单函数写完整四段式注释。下面这些场景可以只写函数名、类型标注和必要的一句话说明：

- 简单 CRUD
- 简单字段转换
- 简单参数校验
- 测试辅助函数
- 很短且意图明显的工具函数

重点需要写清楚的对象包括：仓库扫描、忽略规则、解析降级、chunk 切分、Repo Map、混合检索、重排、Prompt 预算控制、证据绑定、缓存和增量刷新。

### 复杂逻辑内部注释
对于关键的、不直观的处理步骤，必须加中文行内注释，重点解释：

- 为什么这里要这么做
- 如果不这么做会有什么问题
- 这一步是在规避什么风险

简单直白的代码不需要逐行注释，避免注释比代码还多。

### 注释语言与风格
必须统一遵守：

1. 全部使用中文注释
2. 用最通俗的语言写，默认读者是编程基础较弱的人
3. 不要写空话，例如“这里进行处理”“这里调用函数”这种没信息量的话
4. 全项目注释风格统一

## 6.2 代码风格要求

1. 命名必须清晰，不使用难懂缩写  
   例如用 `embedding_provider`，不要写成 `emb_pvd`。

2. 一个函数只做一件事  
   如果一个函数同时负责扫描、过滤、摘要、入库，必须拆开。

3. 函数名必须能完整描述行为  
   例如 `build_repository_summary` 比 `build_summary` 更清楚。

4. 每个模块先写能跑通的最小版本  
   先完成当前模块最核心的能力，再考虑增强逻辑。

5. 不提前写“以后可能用得上”的代码  
   只实现当前验收标准要求的功能，不提前设计用不上的扩展点。

6. 优先写可验证代码  
   每个模块都应该有最小测试办法，确保不是“看起来写完了”，而是真的可运行。

## 6.3 每个模块开发完成后的确认环节

每完成一个模块，并且验证通过之后，必须增加一个“面向用户解释”的确认环节。解释内容必须用通俗中文，包含：

1. 这个模块解决了什么问题
2. 为什么要单独拆成一个文件或一个类
3. 这个模块体现了什么复用价值
4. 涉及了什么新概念（如果用了抽象类、装饰器、依赖注入、缓存等，要讲清楚）
5. 面试官可能怎么问这个模块，用户可以怎么答

要求：

- 在用户能大致复述该模块作用之前，不进入下一个复杂模块
- 回答时优先用类比和自然语言解释，不要一上来堆术语
- 如果用户没有完全理解，先补解释，再继续开发

# 7. 核心模块职责（每个模块必须职责单一）

## 6.1 repo_scanner.py
职责：
- 校验目录是否为 git 仓库
- 扫描目录树
- 应用忽略规则
- 识别文件类型
- 生成文件基础清单

## 6.2 github_adapter.py
职责：
- 只读调用 GitHub API
- 获取仓库元数据
- 获取默认分支
- 获取 Issue / PR 标题与摘要
- 生成 GitHub 文件网页链接

禁止：
- 写入 GitHub
- 提交 comment
- 修改 PR

## 6.3 file_parsers/
职责：
- Markdown 解析
- Python 结构解析
- JS/TS 结构解析
- JSON/YAML/TOML 配置解析

要求：
- 保留标题、函数、类、导出名、路径信息
- 解析失败时要降级成纯文本模式

## 6.4 chunking/
职责：
- 将解析结果切成知识片段
- 片段必须带来源信息
- 代码片段尽量按函数/类边界切
- 文档片段尽量按标题层级切

禁止：
- 粗暴按固定字符数切块

## 6.5 repo_map_builder.py
职责：
- 生成目录树摘要
- 识别入口文件
- 生成模块级地图
- 识别关键目录
- 构建“先导航再检索”的仓库地图

这是大仓库 token 控制的核心模块，必须优先做好。

## 6.6 summary_builder.py
职责：
- 生成仓库整体认识结果
- 输出项目用途、核心价值、技术栈、模块概览、推荐阅读路径、启动说明

## 6.7 embedding_provider.py
职责：
- 统一嵌入模型调用
- 支持本地模型
- 提供批量嵌入接口
- 做失败重试和缓存

## 6.8 vector_store.py
职责：
- 片段入库
- 片段删除
- 相似检索
- 持久化索引

## 6.9 keyword_index.py
职责：
- 建立关键词索引
- 支持路径检索、标题检索、关键字检索

## 6.10 retriever.py
职责：
- 执行混合召回
- 汇总关键词命中与向量命中
- 给出候选片段集

## 6.11 reranker.py
职责：
- 对候选片段进行重排
- 综合路径命中、符号命中、相似度、文件优先级排序

无模型时可简化为规则重排。

## 6.12 prompt_builder.py
职责：
- 将问题、Repo Summary、证据片段、限制说明组织成最终 prompt
- 控制上下文预算

## 6.13 answer_generator.py
职责：
- 调用本地 LLM 生成最终答案
- 要求输出“结论 + 依据 + 不确定点”
- 绑定引用来源

## 6.14 suggestion_engine.py
职责：
- 先做规则检测
- 再让模型把规则结果组织成自然语言建议

不要直接让模型凭空提建议。

## 6.15 jobs.py
职责：
- 管理后台任务：首次索引、增量更新、清理缓存、重建索引
- 提供任务状态和进度

# 7. API 设计（固定契约，先定后写）

所有接口统一 `/api/v1` 前缀。

## 7.0 通用响应约定

所有失败响应统一使用以下结构，前端只按这一种格式展示错误：

```json
{
  "error": {
    "code": "REPO_NOT_FOUND",
    "message": "没有找到指定仓库。",
    "detail": "repo_id=repo_xxx 不存在或已被删除。",
    "trace_id": "trace_xxx"
  }
}
```

任务状态固定为：`queued`、`running`、`succeeded`、`failed`、`cancelled`。

证据项固定结构：

```json
{
  "file_path": "src/main.py",
  "chunk_id": "chunk_xxx",
  "start_line": 10,
  "end_line": 42,
  "source_type": "code",
  "score": 0.82,
  "reason": "路径和问题关键词同时命中",
  "snippet": "def main(): ..."
}
```

`confidence` 固定取值：`high`、`medium`、`low`。当证据不足、命中片段冲突或模型无法确认时，必须返回 `low` 并在回答中说明不确定点。

## 7.1 仓库管理
### POST `/api/v1/repos`
注册仓库。

首版只要求支持本地 `repo_path`。公开 GitHub `remote_url` 自动 clone 是增强能力，可以在 MVP 跑通后再实现。

请求：
```json
{
  "repo_path": "D:/projects/example-repo",
  "remote_url": null,
  "branch": "main",
  "alias": "示例仓库"
}
```

响应：
```json
{
  "repo_id": "repo_xxx",
  "status": "registered",
  "current_commit": "abc123"
}
```

### GET `/api/v1/repos/{repo_id}`
获取仓库基础信息。

---

## 7.2 索引与同步
### POST `/api/v1/repos/{repo_id}/ingest`
触发首次或增量索引。

请求：
```json
{
  "mode": "full",
  "force": false
}
```

响应：
```json
{
  "job_id": "job_xxx",
  "status": "queued"
}
```

### POST `/api/v1/repos/{repo_id}/refresh`
基于当前仓库状态做增量刷新。

### GET `/api/v1/jobs/{job_id}`
查询任务进度。

---

## 7.3 仓库理解
### GET `/api/v1/repos/{repo_id}/summary`
获取仓库概览结果。

### GET `/api/v1/repos/{repo_id}/map`
获取仓库地图。
支持参数：
- `view=tree`
- `view=module`
- `view=dependency`

---

## 7.4 问答与检索
### POST `/api/v1/repos/{repo_id}/qa`
主问答接口。

请求：
```json
{
  "question": "这个项目怎么启动？",
  "top_k": 8,
  "answer_style": "standard",
  "with_evidence": true,
  "with_suggestions": true
}
```

响应：
```json
{
  "answer": "...",
  "evidence": [],
  "suggestions": [],
  "confidence": "medium",
  "used_context": [],
  "trace_id": "trace_xxx",
  "next_steps": []
}
```

### POST `/api/v1/repos/{repo_id}/search`
单独做证据检索。

### GET `/api/v1/repos/{repo_id}/evidence/{trace_id}`
回看某次问答证据链。

---

## 7.5 文件详情
### GET `/api/v1/repos/{repo_id}/files/{file_id}`
获取文件摘要、符号、chunk 数量。

### GET `/api/v1/repos/{repo_id}/chunks/{chunk_id}`
获取单个片段详情。

---

## 7.6 建议
### POST `/api/v1/repos/{repo_id}/suggest`
生成简单建议。

# 8. 大仓库处理策略（必须实现，不可省略）

这是本项目的关键技术设计。

## 8.1 目标
不是让模型“完整理解整个大仓库”，而是让系统通过分层表示在可控成本下回答高频问题。

## 8.2 强制策略

1. **分层索引**  
   - 第一层：目录树 / Repo Map  
   - 第二层：文件摘要  
   - 第三层：chunk 级索引

2. **增量更新**  
   通过文件 hash / mtime / git 快照，只重建变化文件。

3. **热区优先**  
   优先处理：
   - README
   - docs
   - src
   - app
   - config
   - package/requirements/pom/pyproject

4. **忽略无关目录**  
   必须默认忽略：
   - `.git`
   - `node_modules`
   - `dist`
   - `build`
   - `coverage`
   - 压缩包
   - 大二进制文件

5. **大文件降级**  
   超过阈值的大文件：
   - 只做摘要
   - 只做符号提取
   - 不做全量切块

6. **分目录懒加载**  
   当用户第一次问到某个模块时，再生成更细粒度摘要。

# 9. Token 控制规则（必须显式写进实现）

## 9.1 禁止行为
- 禁止整仓长上下文回答
- 禁止每次都重新摘要整个仓库
- 禁止聊天历史全量回放
- 禁止无上限地把命中 chunk 全塞给模型

## 9.2 必须实现的规则

1. **先地图后上下文**  
   先由 Repo Map 缩小范围，再决定调用哪些片段。

2. **摘要缓存**  
   目录摘要、文件摘要、函数摘要必须缓存。

3. **固定上下文预算**  
   一次回答最多带 6~10 个 chunk。

4. **模型分层**  
   - 轻任务：问题分类、query rewrite、重排  
   - 重任务：最终答案生成

5. **检索分级**  
   简单问题先用关键词命中；复杂问题再走向量检索。

6. **答案引用而不是复述整段代码**  
   模型优先引用文件路径、函数名、摘要，避免重复贴源码。

7. **会话裁剪**  
   只保留当前仓库最近几轮和必要摘要。

8. **重复问题缓存**  
   相同仓库版本 + 相似问题可以复用答案缓存。

# 10. 桌面 UI 结构（简单、清晰、可演示）

首版 UI 不要复杂，固定为四页。

## 10.1 RepositoriesPage
职责：
- 添加仓库
- 查看仓库列表
- 显示索引状态
- 触发首次索引 / 增量同步 / 清空索引

## 10.2 ChatPage
职责：
- 显示当前仓库概览卡片
- 输入问题
- 显示回答
- 显示证据列表
- 显示建议列表
- 提供问题模板快捷入口

建议布局：
- 左侧：仓库概览、目录摘要、关键文件
- 中间：聊天问答区
- 右侧：证据与建议面板

## 10.3 LogsPage
职责：
- 查看最近扫描任务
- 查看失败文件
- 查看模型错误
- 查看 GitHub API 错误

## 10.4 SettingsPage
职责：
- 配置 GitHub Token
- 配置本地模型地址
- 配置嵌入模型名
- 配置索引目录
- 配置忽略规则
- 配置最大文件大小

# 11. 开发顺序（严格按顺序，不要跳）

## 第 1 阶段：工程骨架与配置
先做：
1. 创建目录结构
2. 配置桌面端与后端启动脚本
3. 搭建 FastAPI 基础服务
4. 搭建 Electron 基础界面
5. 建立 SQLite 基础表

验收：
- 前后端能本地启动
- 桌面端能调用后端健康检查

## 第 2 阶段：仓库接入与扫描
先做：
1. 仓库注册接口
2. git 仓库校验
3. 文件扫描
4. 忽略规则
5. FileRecord 入库

验收：
- 能添加一个本地仓库
- 能看到扫描后的文件记录

## 第 3 阶段：解析与切块
先做：
1. Markdown 解析
2. Python / JS / TS 基础解析
3. chunk 生成
4. ChunkRecord 入库

验收：
- 能为 README 和几个代码文件生成结构化片段

## 第 4 阶段：Repo Map 与仓库概览
先做：
1. 目录树摘要
2. 关键模块提取
3. 启动方式识别
4. 推荐阅读路径生成
5. RepoSummary 生成

验收：
- 能生成一份仓库概览卡片

## 第 5 阶段：检索层
先做：
1. 关键词索引
2. 向量索引
3. 混合召回
4. 候选重排

验收：
- 给一个问题时，能返回合理的命中片段列表

## 第 6 阶段：问答层
先做：
1. PromptBuilder
2. AnswerGenerator
3. 带引用回答
4. 会话记录

验收：
- 能问“这个项目怎么启动”并返回带文件引用的回答

## 第 7 阶段：建议生成
先做：
1. 规则检测
2. SuggestionEngine
3. 建议展示

验收：
- 能输出至少 2~3 条合理建议

说明：本阶段属于首版后的增强阶段。不要在“添加仓库 → 索引 → 检索 → 带证据问答”跑通之前实现建议生成。

## 第 8 阶段：桌面 UI 联调
先做：
1. 仓库列表页
2. 问答页
3. 证据区
4. 日志页
5. 设置页

验收：
- 用户可以完整走通：添加仓库 → 索引 → 提问 → 看证据

## 第 9 阶段：日志、缓存、增量刷新
先做：
1. 任务状态
2. 缓存层
3. 增量同步
4. 错误展示

验收：
- 重复提问速度明显快于首次问答
- 修改仓库文件后可增量刷新

说明：增量刷新、日志诊断和缓存优化都很重要，但它们不应该阻塞首个演示版本。首版可以先提供重建索引，后续再做精细增量。

# 12. 验收标准（DoD）

项目完成时，必须至少满足：

1. 能添加一个本地 GitHub 仓库目录
2. 能完成首次索引
3. 能生成仓库概览
4. 能对至少 4 类高频问题进行回答：
   - 项目用途
   - 启动方式
   - 模块定位
   - 配置说明
5. 回答中必须带证据
6. 遇到大仓库时不会整仓塞 prompt
7. 桌面端能完成添加仓库、查看仓库概览、提问、查看证据这条主流程
8. 没有 GitHub Token 时仍然可以用本地仓库模式运行
9. 所有关键失败都有中文错误提示

增强版完成时，建议继续满足：

1. 桌面端能查看日志、设置、文件上下文
2. 修改部分文件后可以增量刷新
3. 重复提问可以复用缓存结果或半缓存结果
4. 能基于规则输出有限度改进建议

# 13. 给 AI 执行者的最后说明

如果你是根据这份文档直接实现项目的 AI，请遵守以下原则：

1. 先实现最小闭环，不要一开始追求功能完整
2. 优先保证“仓库接入 → 索引 → 问答 → 证据返回”打通
3. 遇到可选架构决策时，优先选择更轻、更稳定、更容易测试的方案
4. 不要擅自扩大功能边界
5. 不要把桌面端做复杂，桌面端只是载体
6. 不要把多 Agent 做重，重点是检索质量和 token 控制
7. 所有输出优先可验证、可追溯、可解释

一句话总结本手册的实现目标：

> **做一个真正能帮助用户认识陌生 GitHub 仓库的本地知识助手，而不是一个看起来很聪明、实际上没有证据和边界的聊天机器人。**
