# 《面向公开 GitHub 仓库的知识理解与问答助手》项目计划书

**项目代号**：RepoMind  
**项目定位**：课程项目 / 简洁可落地的桌面应用原型  
**项目形态**：轻量桌面端 + 本地后端服务 + RAG 知识库  
**核心目标**：把任意公开 GitHub 仓库快速转化为“可理解、可问答、可总结、可给建议”的知识视图，帮助用户更高效地认识和学习陌生项目。

---

# 一、项目背景与问题定义

对于学生、初中级开发者、课程小组成员来说，面对一个陌生 GitHub 仓库时，通常会遇到几个非常现实的问题：

1. **不知道这个项目到底是干什么的**  
   即使有 README，也可能写得不完整，或者描述过于粗略。

2. **不知道应该从哪里开始看**  
   仓库目录很多，前后端混在一起，入口文件和核心模块不清楚。

3. **不知道项目的核心价值是什么**  
   比如：这个项目解决了什么问题？亮点在哪？值不值得继续学？

4. **不知道如何围绕项目提问**  
   想问“登录逻辑在哪”“这个项目怎么启动”“后端接口在哪定义”，但人工翻仓库很慢。

5. **大仓库很难一次性理解**  
   如果直接把整个仓库都交给大模型，不仅 token 消耗极大，而且很容易失焦、变慢、变贵。

因此，本项目希望解决的问题不是“写一个普通聊天机器人”，而是：

> **如何把任意公开 GitHub 仓库接入为一个临时代码知识库，让系统先形成对仓库的整体认识，再支持用户围绕仓库进行问答、学习和理解。**

---

# 二、项目定位

本项目定位为：

> **一个面向公开 GitHub 仓库的知识理解与问答助手。**

它不是：
- 不是完整 IDE
- 不是自动改代码工具
- 不是企业级代码审计平台
- 不是全自动 PR 机器人

它更像：

> **“把一个 GitHub 仓库临时接入成知识库，然后像问一个已经认真读过这个项目的人一样去问它问题。”**

也就是说，这个系统不仅要能回答问题，还要先对项目形成比较充分的认识，例如：

- 这个项目是做什么的
- 核心价值是什么
- 解决了什么问题
- 技术栈是什么
- 目录结构如何组织
- 关键模块在哪里
- 应该先看什么后看什么
- 哪些地方可能值得优化

---

# 三、目标用户

本项目主要面向以下用户：

1. **想快速读懂开源项目的学生**
2. **刚接手别人仓库、不知道从哪开始看的开发者**
3. **准备做课程汇报、毕设调研、技术分享的人**
4. **GitHub 新手，希望先判断仓库是否值得学习的用户**
5. **希望把仓库整理成“可讲述材料”的老师、助教、社团成员**

---

# 四、项目核心目标

本项目的目标不是做“万能代码理解系统”，而是完成一个**边界清晰、可演示、可落地**的原型。

## 4.1 核心目标

1. **首版支持添加单个本地 Git 仓库目录，后续增强支持公开 GitHub URL 自动 clone**
2. **自动完成仓库基础扫描和知识抽取**
3. **生成仓库整体认识结果**
4. **支持围绕仓库进行自然语言问答**
5. **对大仓库采用可控的 token 节省策略**
6. **以简单桌面应用作为展示载体**

## 4.2 最终希望用户获得的能力

用户接入仓库后，可以直接问：

- 这个项目主要是做什么的？
- 它的核心价值是什么？
- 这个仓库值得学吗？
- 这个项目怎么运行？
- 核心模块有哪些？
- 登录/路由/数据库逻辑在哪？
- 我应该先读哪几个文件？
- 这个项目有哪些地方可以改进？

---

# 五、项目核心价值

本项目的价值不只是“知识问答”，而是**仓库理解 + 问答 + 学习辅助 + 建议生成**的组合能力。

## 5.1 仓库理解
系统先给出：
- 项目做什么
- 项目为什么存在
- 项目的主要组成部分
- 项目的技术栈和结构

## 5.2 项目学习辅助
系统可以帮助用户形成“阅读路线”：
- 先读 README
- 再看入口文件
- 再看核心目录
- 最后看配置和扩展部分

## 5.3 知识问答
用户可以针对仓库提具体问题，系统基于已建立的知识底座和检索结果进行回答。

## 5.4 改进建议
系统可以基于规则 + LLM 总结，提供有限度的建议，例如：
- README 不完整
- 目录组织混乱
- 某些文件过长
- 缺测试目录
- 配置说明不足

# 六、功能边界与范围控制

为了保证项目可落地，必须严格控制范围。

## 6.1 本项目做什么

- 支持**公开 GitHub 仓库**接入
- 支持**单仓库**知识化处理
- 支持**仓库理解结果生成**
- 支持**自然语言问答**
- 支持**简单建议生成**
- 支持**桌面端交互展示**

## 6.2 本项目暂时不做什么

- 不处理私有仓库权限体系
- 不做多人协作/账号系统
- 不做复杂企业知识库接入
- 不做跨多个仓库联合问答
- 不做自动改代码/自动提 PR
- 不做高精度漏洞审计
- 不承诺对所有语言都深度语义理解
- 不承诺对超大仓库做完整全量分析

# 七、产品能力设计

本项目建议分成两层能力。

## 7.1 第一层：仓库整体认识

系统在首次接入仓库后，先输出一份“仓库知识卡片”，至少包括：

1. 项目一句话简介
2. 核心价值总结
3. 适用场景
4. 技术栈识别
5. 目录结构摘要
6. 关键文件定位
7. 主要模块说明
8. 启动方式
9. 推荐阅读顺序
10. 初步建议

## 7.2 第二层：围绕仓库进行问答

系统支持用户继续提问，例如：

- 这个项目怎么启动？
- 哪个文件是主入口？
- 前后端怎么通信？
- 某个模块是做什么的？
- 某个报错应该先检查哪里？
- 如果我要学习这个项目，先看哪些文件？

回答结果不仅要给出文字，还要尽量附带：
- 文件路径
- 相关代码片段
- 不确定性说明
- 下一步建议查看的内容

# 八、最终输出结果设计

本项目最终输出不应只是聊天内容，而应是**结构化 + 可读化**的组合结果。

## 8.1 结构化输出

后端统一返回：

- `repo_summary`：仓库整体摘要
- `answer`：问题回答
- `evidence`：证据文件和片段
- `confidence`：置信度/边界说明
- `next_steps`：建议下一步查看内容
- `suggestions`：改进建议

## 8.2 可读化输出

桌面界面建议展示：

### 左侧
- 仓库概览卡片
- 目录树
- 关键文件列表

### 中间
- 聊天/问答区
- 仓库知识总结区

### 右侧
- 证据来源
- 命中文件
- 改进建议
- 推荐阅读路径

# 九、系统架构设计

本项目的架构原则是：

> **前后端保持轻量，后端以“仓库接入 + 分层索引 + 检索问答”为核心。**

## 9.1 总体架构

```text
桌面应用（Electron + React + TypeScript）
        ↓
本地后端服务（FastAPI）
        ↓
Repo Ingestion / Repo Map / RAG / QA / Suggestion
        ↓
SQLite + Chroma/FAISS
        ↓
LLM Provider
```

## 9.2 为什么采用轻量架构

原因很简单：
- 你是做课程/简历项目，不是做商业平台
- 重点是仓库理解效果，不是分布式系统设计
- 本地桌面 + 本地后端更容易调试和演示
- 不需要复杂登录和部署体系

---

# 十、技术选型建议

## 10.1 桌面端
推荐：
- **Electron + React + TypeScript + Vite**

原因：
- 和执行手册保持一致，避免开发时在 Tauri/Electron 之间摇摆
- 前端生态成熟，资料多，课程项目更容易快速搭建和调试
- 和本地 FastAPI 后端通过 `localhost HTTP(JSON)` 通信简单直接
- 对 Windows 环境更友好，减少 Rust/Tauri 工具链带来的额外不确定性

说明：Tauri 更轻，但它会引入 Rust 工具链和打包环境成本。为了优先完成可演示原型，本项目首版不采用 Tauri。

## 10.2 后端
推荐：
- **FastAPI**

原因：
- Python 在文件处理、RAG、向量库、模型调用方面方便
- API 结构简单清晰
- 后续扩展方便

## 10.3 本地数据库
推荐：
- **SQLite**

存：
- 仓库信息
- 文件信息
- 索引元数据
- 问答历史
- 缓存记录

## 10.4 向量库
推荐：
- **Chroma**（最简单）

备选：
- FAISS（轻，但元数据需要自己处理）

## 10.5 检索方案
推荐：
- **关键词检索 + 向量检索混合**

理由：
- 只用向量检索不稳定
- 只用关键词检索不够智能
- 混合检索更适合代码仓库场景

# 十一、核心模块设计

## 11.1 Repo Ingestion（仓库接入层）

负责：
- 拉取 GitHub 仓库信息
- 读取 README
- 扫描目录树
- 抽取配置文件
- 过滤无关目录（如 `node_modules`、`dist`、`build`）
- 构建基础文件清单

## 11.2 Repo Map（仓库地图层）

负责：
- 生成目录级摘要
- 识别入口文件
- 识别关键目录
- 生成模块级知识卡片
- 建立符号索引表（函数/类/导出名）

这个层非常关键，因为它是**大仓库 token 控制的核心**。

## 11.3 RAG 知识库层

负责：
- 对 README、文档、配置文件、关键代码文件进行切块
- 建立向量索引
- 保存元数据
- 支持基于问题的检索

## 11.4 Query Router（问题路由层）

负责识别问题类型，例如：
- 项目概览类
- 启动运行类
- 模块定位类
- 架构理解类
- 改进建议类

不同问题走不同检索路径，避免无意义地全库扫描。

## 11.5 Answer Composer（回答生成层）

负责：
- 组织证据包
- 调用模型生成回答
- 输出“结论 + 依据 + 不确定点”

## 11.6 Suggestion Engine（建议生成层）

建议采用：
- **规则检测 + LLM 总结**

不要让模型无依据乱提建议。

规则可先覆盖：
- 缺 README
- 缺测试目录
- 目录层次过乱
- 文件过长
- 配置说明不足

---

# 十二、是否采用多 Agent

这个项目**可以保留 Agent 思想，但不建议做成重型多 Agent 系统**。

## 12.1 建议方案：弱多 Agent / 轻编排

可以保留 3 个角色：

### 1. Repository Understanding Agent
负责形成仓库整体认识：
- 项目做什么
- 核心价值
- 技术栈
- 关键模块

### 2. QA Agent
负责围绕用户问题组织检索并回答。

### 3. Suggestion Agent
负责把规则检测结果组织为自然语言建议。

## 12.2 为什么不建议做重型多 Agent

因为这个项目的核心难点其实是：
- 仓库扫描质量
- 检索质量
- 上下文压缩
- token 成本控制

而不是“Agent 之间如何复杂协作”。

所以建议：

> **多 Agent 作为架构组织方式保留，但实现上以单主流程控制为主。**

# 十三、大仓库 token 消耗问题：核心解决方案

这是这个项目里最重要的技术问题之一。

## 13.1 绝对不能做的事情

不能：
- 把整个仓库全塞进 prompt
- 每次提问都重新全量扫描
- 每次回答都把所有历史上下文回放
- 对所有文件都做同等深度处理

## 13.2 正确策略：分层理解

### 第一层：仓库地图
只看：
- README
- 目录树
- package.json / requirements.txt / Cargo.toml 等
- 入口文件候选
- 核心目录

这一层先回答：
- 这是个什么项目
- 哪些目录重要
- 项目怎么组织

### 第二层：模块摘要
对关键目录和关键文件生成摘要：
- 每个目录一句话
- 每个核心文件一句话
- 每个模块主要职责

### 第三层：按问题检索
只有用户问到某部分时，才进一步拉相关代码块进上下文。

也就是说：

> **先做“地图”，再做“局部深入”，而不是全量理解。**

## 13.3 Token 控制具体措施

1. **摘要缓存**  
   README 摘要、目录摘要、文件摘要一次生成，多次复用。

2. **上下文预算固定化**  
   每次回答只带少量 Top-K chunk，例如 6~10 个。

3. **小模型做轻任务**  
   问题分类、query rewrite、重排用小模型；主模型只负责最终回答。

4. **混合检索缩小范围**  
   先关键词，再向量检索，再重排。

5. **跳过无关目录**  
   默认不处理：
   - `node_modules`
   - `dist`
   - `build`
   - 二进制文件
   - 大图片

6. **大文件降级**  
   超大文件只做摘要和符号索引，不做全量切块。

7. **历史对话裁剪**  
   只保留最近几轮和当前问题相关摘要。

8. **缓存重复问题**  
   相同仓库版本 + 相似问题，可直接返回缓存结果或半缓存结果。

# 十四、MVP 版本范围

为了保证项目能做完，我建议 MVP 只做以下内容：

## 14.1 MVP 必做

1. 添加本地 Git 仓库目录
2. 自动扫描仓库基础信息和文件清单
3. 解析 README、Markdown、Python、JavaScript、TypeScript 和常见配置文件
4. 生成基础 chunk、Repo Map 和仓库概览卡片
5. 建立 SQLite 元数据、SQLite FTS 关键词索引和 Chroma 本地向量索引
6. 提供基础问答
7. 返回文件路径、行号、片段类型和命中原因等证据
8. 提供三栏桌面应用界面：仓库列表 / 问答区 / 证据区

## 14.2 MVP 暂时不做

1. 公开 GitHub URL 自动 clone
2. GitHub Issue / PR 信息接入
3. 多轮深记忆
4. 多仓库联合问答
5. 版本差异比较
6. 自动修复代码
7. 复杂调用链图谱
8. 私有仓库接入
9. 企业级权限管理
10. 完整日志诊断页、设置页、问题模板和建议生成

# 十六、评估方式

这个项目不能只看“模型会不会说话”，必须看是否真的有帮助。

## 16.1 评估维度

### 1. 有用性
用户能不能根据回答：
- 找到关键文件
- 启动项目
- 理解项目结构
- 决定先学什么

### 2. 准确性
回答是否能被仓库内容验证：
- 路径对不对
- 命令对不对
- 模块职责是否合理

### 3. 覆盖度
是否覆盖：
- 项目概览
- 技术栈
- 结构说明
- 问答
- 建议

### 4. 鲁棒性
当仓库文档不全时，系统能否明确表达不确定，而不是胡编。

# 十七、风险与应对

## 风险 1：仓库太大，token 和速度爆炸
**应对**：分层索引、摘要缓存、问题路由、局部检索。

## 风险 2：回答看起来像普通聊天机器人
**应对**：必须输出证据文件和来源片段。

## 风险 3：建议功能容易乱说
**应对**：先做规则检测，再让模型组织语言。

## 风险 4：支持语言太多导致失控
**应对**：MVP 先聚焦 Python + JavaScript/TypeScript。

## 风险 5：桌面端做太复杂拖慢进度
**应对**：界面保持三栏结构，不做复杂协作功能。

# 十八、最终项目一句话总结

你可以把这个项目最终总结成：

> **本项目是一个面向公开 GitHub 仓库的知识理解与问答助手。系统通过仓库扫描、分层摘要、检索增强和轻量 Agent 化编排，将陌生代码仓库转化为可理解、可问答、可总结的知识视图，帮助用户更快认识项目、理解结构、定位问题，并获得初步学习建议与改进建议。**

# 十九、答辩/汇报时的标准说法

如果别人问：

## 1. 这个项目到底是做什么的？
回答：
> 它是一个面向 GitHub 仓库的知识理解与问答助手，可以把公开仓库转化成临时代码知识库，帮助用户快速理解项目并围绕它进行问答。

## 2. 它和普通知识库问答有什么区别？
回答：
> 它处理的不只是普通文档，而是代码仓库这种结构化知识源。除了问答，它还会输出仓库概览、核心模块、推荐阅读路径和改进建议。

## 3. 大仓库怎么办？
回答：
> 不会把整个仓库直接塞给模型，而是先生成 Repo Map 和分层摘要，再按问题局部检索相关内容，从而控制 token 和成本。

## 4. 为什么前后端做得比较简单？
回答：
> 因为本项目的重点在仓库理解和问答能力，而不是做复杂平台。前后端采用轻量架构，更适合课程项目和演示场景。
